
# Компиляция и линковка

## Задача 1: Определения терминов

1. **Опкод (Opcode)** — часть машинной инструкции, определяющая операцию, которую должен выполнить процессор (например, сложение, переход или копирование данных).
2. **Объектный файл (.o)** — файл, содержащий скомпилированный машинный код, данные и таблицу символов, но еще не объединенный в готовую программу.
3. **Линковщик (Linker)** — программа, которая собирает несколько объектных файлов и библиотек в один исполняемый файл, разрешая зависимости между символами.
4. **Статическая библиотека (.a/.lib)** — архив из нескольких объектных файлов, код которых целиком копируется в исполняемый файл на этапе линковки.
5. **Динамический загрузчик (Dynamic Linker)** — системный компонент, который находит и загружает необходимые разделяемые библиотеки (`.so` / `.dll`) в память при запуске или во время работы программы.

## Задача 2: Анализ утверждений

### Разбор ошибок в остальных пунктах

1. **Препроцессор...** — **Неверно**. Препроцессор работает только с текстом: раскрывает макросы, подключает заголовочные файлы (`#include`) и удаляет комментарии. В машинные инструкции код превращает компилятор и ассемблер.
2. **Объектные файлы можно запускать...** — **Неверно**. В объектном файле не разрешены адреса внешних функций и отсутствуют инструкции для операционной системы о том, как правильно запустить процесс (точка входа).
3. **Статическая линковка добавляет код...** - **Верно**. Весь код полностью зашивается в бинарник
4. **ELF файл всегда занимает одинаковое кол-во памяти...** — **Неверно**. Размер ELF-файла напрямую зависит от количества кода, данных и отладочной информации.
5. **.bss содержит инициализированные переменные...** — **Неверно**. Секция `.bss` зарезервирована для **неинициализированных** глобальных и статических переменных (тех, что по умолчанию равны нулю). Инициализированные переменные хранятся в секции `.data`.

## Задача 4: ELF и динамический загрузчик

Для анализа использовался скомпилированный файл `myprog`, утилиты `ldd` и `readelf`.
Что бы увилеть результат можно воспользоватся скриптом `ex4.sh`

### 1. Библиотека для printf

При выполнении команды `ldd myprog` была найдена следующая зависимость:

- **Библиотека:** `libc.so.6`
- **Путь:** `/usr/lib/libc.so.6` (или `/lib/x86_64-linux-gnu/libc.so.6` в зависимости от дистрибутива).
- **Описание:** Именно в этой стандартной библиотеке Си реализована функция `printf`.

### 2. Секция .interp

С помощью команды `readelf -p .interp myprog` было определено содержимое секции:

- **Файл:** `/lib64/ld-linux-x86-64.so.2`
- **Описание:** В этой секции указан путь к динамическому линковщику, который ОС должна запустить для подготовки программы к работе.

### 3. Роль ld-linux.so

Файл `ld-linux-x86-64.so.2` является динамическим загрузчиком. Его основные функции при запуске программы:

1. **Загрузка зависимостей:** Он считывает список необходимых библиотек (из вывода `ldd`) и отображает их в адресное пространство процесса.
2. **Разрешение символов (Relocation):** Он связывает абстрактные вызовы функций в коде (например, вызов `printf`) с конкретными адресами внутри загруженной библиотеки `libc`.
3. **Инициализация:** Он выполняет функции инициализации библиотек и затем передает управление на точку входа программы (`main`).

## Задача 5: Анализ ELF

Для изучения структуры сегментов использовалась команда `readelf -S myprog`.

### 1. Секция для кода программы

Машинные инструкции (скомпилированный код) хранятся в секции **`.text`**. В таблице разделов она помечена флагами `AX` (Alloc, Executable), что означает выделение памяти при запуске и разрешение на выполнение.

### 2. Секции для глобальных переменных

Глобальные переменные распределяются по двум секциям в зависимости от наличия начального значения:

- **`.data`** — для инициализированных переменных (например, `int a = 5;`).
- **`.bss`** — для неинициализированных или инициализированных нулем переменных (например, `int b;`).

### 3. Отличия .data от .bss

- **Размер на диске:** Секция `.data` хранит реальные значения, поэтому она увеличивает размер исполняемого файла. Секция `.bss` (Block Started by Symbol) на диске почти ничего не занимает — в ELF-файле хранится только её суммарный размер.
- **Запуск программы:** При старте процесса операционная система выделяет под `.bss` память в ОЗУ и автоматически заполняет её нулями.

### 4. Права доступа секции .text (READ | EXEC)

Секция с кодом обычно имеет права **только на чтение и выполнение** (флаги `R` и `E`) по следующим причинам:

1. **Безопасность:** Запрет на запись (`W`) предотвращает атаку типа "Self-Modifying Code", не позволяя вредоносному ПО или ошибке в программе перезаписать инструкции на лету.
2. **Совместное использование:** Несколько запущенных процессов одной программы могут использовать одну и ту же область физической памяти с кодом, так как гарантируется, что код не будет изменен.
3. **Предотвращение ошибок:** Попытка программы случайно записать данные в область кода вызовет ошибку `Segmentation Fault`, что облегчает отладку.
