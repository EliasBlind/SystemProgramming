# Отчет по домашней работе: Память, гонки и мьютексы

## Часть 1. Заполнение пропусков

1. У каждой программы есть собственное **адресное пространство**, которое делится на сегменты.
2. Стек работает по принципу **LIFO (Last In, First Out)**.
3. В куче память выделяется с помощью функции **malloc**, а освобождается через **free**.
4. Ошибка обращения к чужой памяти в Linux вызывает сигнал **SIGSEGV (Segmentation Fault)**.
5. Когда два или больше потоков одновременно обращаются к одним и тем же данным, возникает **состояние гонки (race condition)**.
6. Чтобы защитить общий ресурс, используется механизм **мьютекс (mutex)**.
7. Если поток захватывает мьютекс и больше никогда его не отпускает, возникает ситуация **взаимной блокировки (deadlock)**.

---

## Часть 2. Ответы на вопросы

1. **Стек vs Куча:** Стек автоматический и быстрый (для локальных переменных), куча — ручная и гибкая (для динамических данных).
2. **Забытый free():** Произойдет утечка памяти. Она останется занятой до конца работы программы.
3. **«Баги по настроению»:** Потоки переключаются ОС непредсказуемо. Ошибка проявляется только при определенном порядке их выполнения.
4. **pthread_mutex_lock:** Захватывает мьютекс. Если он занят, поток ждет его освобождения.
5. **Автоматика в ОС:** Это было бы слишком медленно. Программист лучше знает, какие данные нужно защищать.
6. **Общая память:** Это позволяет избежать дорогого копирования данных между потоками.
7. **Без стека:** Нет, он критически важен для работы функций и хранения адресов возврата.
8. **Ручной malloc:** Дает максимальный контроль и производительность, необходимые системному софту.
9. **Гарантия отсутствия deadlock:** Нельзя гарантировать простым наличием двух мьютексов. Если один поток ждет мьютекс 1, а другой — мьютекс 2, они могут заблокировать друг друга.

## Часть 3. Ошибки в коде

1. **Утечка памяти:** `malloc` вызывается в каждой итерации цикла, перезаписывая указатель `arr`. Предыдущая память не освобождается.
2. **Использование неинициализированных имен:** Переменная `terr` не объявлена (в коде была `arr`).
3. **Ошибки освобождения:** Лишний вызов `free` и опечатка в `freeterrr`.
4. **Типы данных:** Отсутствие проверки на `NULL` после `malloc`.

## Часть 4. Анализ выполнения

* **Что будет происходить:** Программа будет быстро потреблять память. При 20 итерациях это незаметно, но при миллионе — память закончится. Также программа упадет (Segfault) на строке `free(terr)` из-за неверных имен.
* **Как исправить:** Перенести `free` внутрь цикла или выделить память один раз вне цикла.

1. **Правда.** При нехватке памяти `malloc` возвращает 0 (NULL).
2. **Правда.** Стандарт языка разрешает делать `free(NULL)`.
3. **Правда.** Современные ОС забирают всю память процесса после его закрытия.
4. **Ложь.** Перезапуск процесса лишь временно убирает симптом, но не исправляет ошибку в алгоритме.
